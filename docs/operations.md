# عمليات RangeBot على Windows

## مبادئ التشغيل

- يعمل المحرك كخدمة Windows مستقلة عن المتصفح وRDP.
- يستمع FastAPI والواجهة على `127.0.0.1:8765` فقط.
- تبدأ خدمة RangeBot في `Paper` كحالة آمنة، ثم تُفعّل `Testnet` أو `Live` من مسار تبديل البيئة المخصص داخل التطبيق.
- لا تُعتبر قيمة البيئة المحفوظة مفعّلة حتى يطابقها وضع المحرك وGate adapter وREST وWebSocket وCredential Profile الفعلي.
- تظل الحماية الحقيقية إلزامية: الاعتمادات، المصالحة، حداثة البيانات، حدود المخاطر، الرصيد، قواعد العقد، حماية TP/SL، والإيقاف الطارئ.
- تستخدم البيانات المتغيرة قاعدة SQLite تحت `%LOCALAPPDATA%\RangeBot\data\rangebot.db` ولا تُكتب في مجلد التثبيت.
- يمكن نقل جذر البيانات بالكامل إلى قرص آخر مثل `D:\RangeBotData` عبر
  `RANGEBOT_HOME`. المسارات التي تحتوي مسافات مدعومة، مثل
  `D:\RangeBot Data`. يشمل ذلك قاعدة SQLite والنسخ الاحتياطية والسجلات والاعتمادات
  المحمية وملفات التشغيل المؤقتة.

## تبديل بيئة التداول

تعرض الواجهة البيئة الفعلية من `GET /v1/runtime/environment`، وليس من تفضيل مخزن فقط. يتم التبديل عبر `POST /v1/runtime/environment/switch`، ويشمل Gate adapter وREST وPublic/Private WebSocket وCredential Profile ولقطات المصالحة وبيانات السوق معاً.

سياسة بيانات السوق العامة واضحة ومقصودة: يستخدم `Testnet` عناوين Gate Testnet العامة فقط، ويستخدم `Live` عناوين Gate Live العامة فقط. يستخدم `Paper` أسعار وبيانات السوق العامة من Live لتوفير سوق واقعي، لكنه لا يحمّل Live Credentials ولا Account state ولا يرسل أي Live Order. عند كل تبديل تُعاد تهيئة `GatePublicMarketProvider` و`GateContractRulesProvider` و`GateHistoricalMarketDataProvider` مع البيئة العامة الصحيحة قبل السماح بالعمليات الجديدة.

أثناء التبديل:

1. تتوقف معاينة Orders وإرسالها.
2. تتوقف اتصالات البيئة السابقة.
3. تُبطل لقطات الحساب القديمة وتُمسح بيانات السوق داخل الذاكرة.
4. تُحمّل مكونات وCredentials البيئة المطلوبة فقط.
5. تُجرى مصالحة جديدة قبل اعتماد Gate adapter الجديد.
6. يبقى التداول محظوراً إذا فشل التبديل أو تطلب إعادة تشغيل.

يتطلب الانتقال إلى `Live` تأكيد `SWITCH TO LIVE`. لا يسمح التبديل بوجود Strategy Instance نشطة أو Position أو Order مفتوح أو حالة Gate.io غير مُدارة. اختيار `Live` أو `Testnet` داخل شاشة Credentials يغيّر Credential Profile الجاري تحريره فقط، ولا يغيّر بيئة التداول.

يُحفظ سجل تفعيل مستقل فقط بعد نجاح التبديل الكامل، وتستعيده الخدمة عند إعادة التشغيل. لا تُعامل قيم `environment` القديمة في إعدادات الإصدارات السابقة على أنها دليل تفعيل؛ بعد الترقية يبدأ المحرك في `Paper` إلى أن ينفذ المستخدم أول تبديل ناجح عبر المسار الجديد.

## Reconciliation وPreview

لا تنفذ Preview مصالحة Gate.io كاملة داخل طلب المستخدم. يحتفظ `ReconciliationCoordinator` بلقطة حساب محدودة العمر، ويطلب التحديث في الخلفية مع single-flight لمنع تكرار الطلب نفسه، وثلاث محاولات بــ backoff، ومهلة محدودة للطلب اليدوي. إذا كانت اللقطة مفقودة أو قديمة أو فشلت المزامنة، تعود Preview بسرعة مع `can_submit=false` وسبب دقيق مثل `reconciliation_snapshot_missing` أو `reconciliation_snapshot_stale` أو `reconciliation_timeout`. لا يتحول نقص البيانات إلى رسالة خاطئة عن بلوغ حد المخاطر.

يمكن فحص الحالة المنقحة عبر `GET /v1/exchange/{mode}/reconciliation` وطلب تحديث محدود عبر `POST /v1/exchange/{mode}/reconcile`. تبقى Credentials، تطابق البيئة، حداثة السوق، الحماية، الرصيد، وقواعد العقد حواجز مستقلة.

## Daily Risk Policy

تحتفظ RangeBot بخط أساس Equity واحد غير قابل للتعديل لكل بيئة ولكل يوم حسب `Asia/Riyadh`. يُلتقط أول Equity point موثوق لذلك اليوم مرة واحدة؛ لا يُرفع الخط بعد خسارة ولا يُعاد ضبطه عند Recovery أو Restart. يبدأ يوم جديد بسجل Baseline جديد مستقل، وتبقى سجلات Testnet وLive منفصلة.

لكل Daily Limit حالة `enabled` صريحة في SQLite: Daily Equity-Loss، Daily Losing-Trades، وDaily Automatic-Entry. تعطيل Limit لا يستخدم أرقاماً ضخمة ولا يغير قيمته المحفوظة، بل يجعله يظهر كـ `disabled`. الحالات الأخرى مستقلة وواضحة: `not_reached` و`reached` و`data_unavailable` و`synchronizing`. لا تُعرض رسالة Limit reached عند غياب Baseline أو Account data أو أثناء Reconciliation.

سياسة Limits مشتركة بين Testnet وLIVE، لذلك يتطلب إضعاف أي Limit مفعّل العبارة `DISABLE LIVE RISK LIMITS` حتى عند فتح الإعدادات من Paper أو Testnet. هذا يمنع تجاوز تأكيد الأموال الحقيقية عبر تبديل البيئة أولاً. ولا يتجاوز التعطيل Environment Matching أو Credentials أو Fresh Account Data أو sufficient Balance أو valid Quantity أو unmanaged-state protection أو Protection-Order validation.

## Paper and Testnet verification

تشغيل دورة Paper/Testnet الآلية وخطوات Gate.io Testnet اليدوية الآمنة موثقة في [`PAPER_TESTNET_VERIFICATION.md`](PAPER_TESTNET_VERIFICATION.md). لا تستخدم هذه العملية LIVE credentials أو LIVE Orders أو أموالاً حقيقية.

## نقل بيانات RangeBot إلى القرص D بأمان

1. فعّل Emergency Stop وأوقف أي بوتات نشطة، ثم أوقف خدمة `RangeBotEngine`.
2. أنشئ نسخة احتياطية من داخل التطبيق وتحقق من صلاحيتها.
3. انسخ مجلد `%LOCALAPPDATA%\RangeBot` كاملاً إلى مسار مطلق، مثلاً
   `D:\RangeBotData`. لا تنقل ملف قاعدة البيانات منفرداً.
4. شغّل `deploy\install-service.ps1` بصلاحية مسؤول مع مسار التثبيت الحالي
   و`-DataRoot 'D:\RangeBotData'`. يحتفظ المثبت الآن بالمسار الصريح خارج
   `AppData` ويكتبه في متغير خدمة WinSW `RANGEBOT_HOME`.
5. ابدأ الخدمة وتحقق من `/health`، ثم نفّذ Refresh Reconciliation وتأكد من
   المراكز والأوامر والحماية قبل السماح بأي دخول جديد.
6. احتفظ بالنسخة القديمة حتى نجاح الترحيلات والمصالحة والنسخ الاحتياطي الجديد.

للتطوير والاختبارات يمكن أيضاً تعيين `TEMP` و`TMP` إلى مجلد على D لتجنب إنشاء
قواعد SQLite المؤقتة على قرص النظام.
- لا تُوزع ملفات `.env` ولا قواعد بيانات مطور ولا سجلات ولا نسخ احتياطية داخل المثبت.

## التثبيت والتحديث

استخدم `release\RangeBot-Setup.exe` فقط للمستخدم النهائي. يثبت البرنامج:

- المحرك المجمّع بدون نافذة Console.
- `RangeBot.exe` لفتح لوحة localhost.
- WinSW وخدمة `RangeBotEngine` ذات بدء تلقائي واستعادة بعد الأعطال.
- لوحة React المجمعة والترحيلات واعتماديات التشغيل.

يشغّل المثبت الخدمة بحساب Windows المدمج `LocalService` من دون طلب كلمة مرور. يحدد مثبت الخدمة مستخدم Explorer النشط في جلسة Windows أو RDP نفسها، ويستخرج مسار ملفه الشخصي حتى يبقى جذر البيانات `%LOCALAPPDATA%\RangeBot` تابعاً للمستخدم الأصلي حتى عند إدخال اعتماد مسؤول مختلف في نافذة UAC. تُمنح الصلاحية لذلك المستخدم وLocalService وSYSTEM والمسؤولين المحليين، وتُشفّر الاعتمادات باستخدام DPAPI بنطاق الجهاز. عند التحديث تتوقف الخدمة وتُحدّث ثم تبدأ من جديد من دون تغيير مسار البيانات.

لفتح البرنامج، شغّل `RangeBot.exe`. يتحقق المشغل من صحة المحرك، ويحاول بدء أو إعادة تشغيل الخدمة، ثم ينتظر الجاهزية ويفتح `http://127.0.0.1:8765/app/`.

## النسخ الاحتياطية

نفّذ عمليات النسخ من صفحة **النسخ الاحتياطية والسجلات** أو من واجهة localhost API. لا تستخدم أدوات قاعدة بيانات خارجية.

تدعم الواجهة:

- إنشاء نسخة SQLite سليمة.
- عرض أحدث النسخ.
- حذف نسخة محددة.
- استعادة نسخة بعد تأكيد صريح.

تحتفظ RangeBot بأحدث عشر نسخ تحت `%LOCALAPPDATA%\RangeBot\backup`.

### الاستعادة الآمنة

قبل أي تغيير في الحالة، يتحقق المحرك من أن الملف:

1. موجود داخل مجلد نسخ RangeBot فقط.
2. قاعدة SQLite سليمة عبر `PRAGMA quick_check`.
3. يحتوي جدول ترحيلات RangeBot.

بعد التحقق:

1. يوقف الاستراتيجيات العاملة والمراقِبة.
2. يفعّل Emergency Stop في Testnet وLive.
3. ينشئ نسخة أمان من الحالة الحالية.
4. يستبدل القاعدة ذرّياً.
5. يشغل Alembic migrations.
6. يبطل لقطات المصالحة القديمة.
7. يعيد مصالحة وضع Gate.io المهيأ.
8. يبقي Emergency Stop نشطاً حتى مراجعة المستخدم يدوياً.

لا تعيد الاستعادة التحويل إلى Paper ولا تزيل Emergency Stop تلقائياً.

## سجلات الدعم

تُحفظ سجلات WinSW الدوّارة تحت `%LOCALAPPDATA%\RangeBot\logs`.

زر **تصدير سجلات الدعم** ينشئ ZIP محدود الحجم ويتضمن ملفات نصية مناسبة فقط. يستبعد قواعد البيانات والنسخ الاحتياطية والاعتمادات والمفاتيح الخاصة وأسماء الملفات الحساسة والتصديرات السابقة، وينقح رؤوس المصادقة والقيم السرية الشائعة.

## فحوصات Windows المطلوبة قبل إصدار نهائي

- تثبيت `RangeBot-Setup.exe` على حساب Windows نظيف بلا Python أو Node.js.
- التحقق من بدء `RangeBotEngine` تلقائياً ومن دون Console.
- التحقق من بقاء المحرك بعد إغلاق المتصفح وفصل RDP.
- إعادة تشغيل Windows والتحقق من استعادة الحالة والمصالحة قبل أي دخول.
- التحقق من DPAPI تحت حساب الخدمة الحقيقي.
- اختبار إنشاء/استعادة نسخة من واجهة المستخدم على Windows.
- اختبار خيار إلغاء التثبيت مع الاحتفاظ بالبيانات وخيار حذفها الصريح.

سجّل الأوامر والنتائج ولقطات الشاشة الفعلية. لا تعتبر ملف الإعداد أو اختبارات المصدر دليلاً بديلاً عن تثبيت Windows حقيقي.
